Database Design

Entity-Relationship model, table schemas, column constraints, relationships, and indexes for MemoLink's PostgreSQL database with pgvector.

Smart Source Workspace Tables

source_files stores OneDrive identifiers, file metadata, extraction state, and version metadata. book_note_links links a book, source, and editable note. file_annotations stores editable normalized stroke/highlight/comment JSON. Stroke payloads retain pointer type, pen type, opacity, normalized points, timestamps, pressure, and available tilt; colour, pen size, tool type, page, and location anchor remain queryable columns. note_timeline_events records source-workspace actions. recording_metadata stores local-only filename and duration metadata.

These tables intentionally contain no original file, book, recording, Blob, or base64 binary columns. Source binaries remain in OneDrive; recording binaries remain local.

Negative temporary annotation IDs and save queues exist only in frontend memory. The database assigns canonical positive IDs. Partial erasing updates/splits stroke JSON; whole-stroke erasing uses the existing soft-delete lifecycle so synchronized clients retain an auditable model without altering the original source.

ERD

Entity-Relationship Overview

MemoLink uses nine original core tables - users, workspaces, notes, embeddings, conversations, messages, reminders, feature_flags, and feedback - plus additional extension tables for system logs, translation cache, API keys, Gmail integration, graph memory, proactive insights, evaluation analytics, survey responses, and the Books Library. All user-scoped tables cascade-delete when the user is removed. Notes, conversations, and workspaces carry a deleted_at column for soft-delete support. Notes, conversations, and reminders each carry a workspace_id foreign key so every piece of content is scoped to a workspace. Embeddings remain linked to notes; retrieval is workspace-scoped by joining through the note's workspace_id.

New tables for the Books Library: books stores the OneDrive-synced catalogue; user_books tracks per-user borrowing and reading progress; book_bookmarks and book_highlights store per-user annotations, with each highlight mirrored into a Note; book_note_sources and book_note_chunks back the async "Save as Note Source" extraction job; onedrive_accounts stores one encrypted Microsoft OAuth token pair per connected admin.

Schema stability for most new features: AI image generation (with fallback chain), multi-model chat, translation, web search, agent mode, and password reset are all stateless at the database layer - they require no additional tables or columns. Generated images are embedded as base64 data URLs inside the existing messages.content TEXT column (base64 is stripped from context before OpenAI calls to prevent token overflow). Model attribution is carried in the SSE stream and rendered client-side. Rich HTML from bulk file upload is stored in notes.content, which already accepts arbitrary HTML including embedded <img> tags.

New tables for admin features: The feature_flags table stores system-wide feature toggles, defaults, and per-feature minimum access levels. The feedback table stores user-submitted bug reports and suggestions. The users table gains an is_admin BOOLEAN column and an access_level VARCHAR column ('regular' | 'plus' | 'pro'). JWT tokens embed both is_admin and access_level claims; the get_current_user_info dependency decodes them for stateless access-level enforcement without a DB lookup on every request.

Table

users

Stores registered user accounts. The root entity - all other tables are scoped to a user via foreign keys. The is_admin column controls access to admin-only API endpoints; the first registered user is auto-promoted to admin on application startup. The access_level column ('regular' / 'plus' / 'pro') gates premium features; admins can change any user's level via PATCH /api/admin/users/{id}/level. Both is_admin and access_level are embedded in the JWT so feature enforcement is stateless.

👤 users PostgreSQL table
ColumnTypeConstraintsNotes
id INTEGER PKINDEX Auto-incrementing primary key
email VARCHAR UNIQUENOT NULL Unique email address used for login
password VARCHAR NOT NULL Hashed password (bcrypt via passlib)
is_admin BOOLEAN default: false Admin flag - when true, the user can access admin-only API endpoints and the Admin Panel UI. The first registered user is promoted to admin automatically on startup. Admins can promote or demote other users via PATCH /api/admin/users/{id}/role. Admin status bypasses all access-level checks. Added via ALTER TABLE IF NOT EXISTS at startup.
access_level VARCHAR NOT NULL default: 'regular' Subscription tier: 'regular', 'plus', or 'pro'. Controls which feature-flag-gated features the user can access. Admins change it via PATCH /api/admin/users/{id}/level. Embedded in the JWT on login - level changes require the user to re-login to take effect. Added via ALTER TABLE users ADD COLUMN IF NOT EXISTS access_level VARCHAR NOT NULL DEFAULT 'regular' at startup.
Table

workspaces

Organises a user's knowledge into named, typed containers. Every note, conversation, and reminder belongs to a workspace. Workspaces are soft-deleted via deleted_at - a user must always have at least one non-deleted workspace. The is_default flag marks the workspace created during first-time onboarding. last_accessed_at is updated each time the user activates the workspace and is used to auto-select it on the next login.

🗂️ workspaces PostgreSQL table - created at startup via ALTER TABLE IF NOT EXISTS

name VARCHAR(100) NOT NULLUNIQUE per user Display label - e.g. "Capstone Research", "Claims System". Unique within the user's workspaces (enforced by composite unique index on user_id + name).
type VARCHAR(30) NOT NULL One of: Academic, Professional, Personal, Project, Other
description TEXT nullable Optional freeform description - shown in the workspace management panel
is_default BOOLEAN default: false Marks the workspace created during first-time onboarding. Only one workspace per user should have this set to true at any time.
last_accessed_at TIMESTAMPTZ nullable Updated each time the user activates this workspace. Used to restore the last active workspace on next login.

deleted_at TIMESTAMPTZ nullable Soft-delete timestamp - non-null means the workspace is hidden. The API blocks deletion if this would leave the user with zero active workspaces.

notes

Stores user-created notes. Each note is owned by one user, scoped to one workspace, and has at most one associated embedding (1-to-1). Soft-deleted rows have a non-null deleted_at and are excluded from retrieval. The RAG pipeline filters embeddings through this table's workspace_id to enforce workspace-scoped retrieval.

Table

messages

Stores individual chat messages (both user and AI) within a conversation. Supports paginated retrieval via before_id cursor.

Content format note: The content column stores free-form text which may contain Markdown, LaTeX, or - for AI image generation responses - a Markdown image tag embedding a base64 data URL (e.g. ![image](data:image/png;base64,...)). Base64-embedded images are persistent and do not expire. For AI image replies, the content also includes DALL-E's revised prompt appended after the image. Translation responses are stored as plain text. No schema change is required for these content types - the TEXT column accommodates them without length restriction.

📨 messages PostgreSQL table
ColumnTypeConstraintsNotes
id INTEGER PK Auto-incrementing primary key
conversation_id INTEGER FK → conversations.id CASCADE DELETE - messages deleted with conversation
role VARCHAR(20) Either "user" or "assistant"
content TEXT Message body - Markdown and LaTeX supported. AI image generation replies include a Markdown ![image](data:image/...;base64,...) tag with an embedded base64 data URL. Translation responses are stored as plain text. All content types are handled by the TEXT column without schema changes.
created_at TIMESTAMPTZ server default: now() Used for ordering and paginated cursor (before_id)

system_logs

Stores structured application events at three severity levels - INFO, WARNING, and ERROR. Written by the SystemLogService throughout the application (auth, transcription, translation, chat fallbacks, uploads). The table has no foreign keys; user_id is stored as a plain integer so log entries are preserved even after a user is deleted. Admins view, filter, paginate, and clear logs via the System Logs tab in the Admin Panel.

Table

translation_cache

Caches successful translation results to avoid redundant LLM calls. The cache key is the SHA-256 hash of {source_text}::{target_language.lower()}. On any subsequent translate request for the same text and language, the cached result is returned immediately with cached: true and the hit_count is incremented. A force: true field bypasses the cache, calls the LLM pipeline, and upserts the result. The table has no foreign keys and is not scoped to any user - all users share the same cache pool.

⚡ translation_cache PostgreSQL table - created at startup via CREATE TABLE IF NOT EXISTS
ColumnTypeConstraintsNotes
id SERIAL PK Auto-incrementing primary key
text_hash VARCHAR(64) UNIQUENOT NULLINDEX SHA-256 hex digest of {source_text}::{target_language.lower()}. Primary lookup key - unique constraint ensures one cache entry per text+language combination.
source_text TEXT NOT NULL The original text that was translated. Stored for debugging and re-translation reference.
target_language VARCHAR(100) NOT NULL The target language name as submitted - e.g. "Māori", "Japanese", "Tagalog". Lowercased when building the hash key.
translation TEXT NOT NULL The cached translation result returned to the client when a cache hit occurs.
accuracy INTEGER nullable Semantic similarity score 0-100 from the back-translation scoring round. null when scoring was skipped (e.g. GPT fallback path where no scoring round runs).
model VARCHAR(100) NOT NULL The model that produced this translation - e.g. "gemini-2.0-flash-lite", "gpt-4o-mini". Preserved across cache hits for display in the UI attribution badge.
hit_count INTEGER NOT NULL default: 0 Incremented each time this cache entry is served to a user. Useful for analytics and identifying heavily-used translations.
created_at TIMESTAMPTZ NOT NULL default: now() Timestamp when this entry was first written (UTC).
updated_at TIMESTAMPTZ NOT NULL default: now() Refreshed on upsert - i.e. when a force: true re-translate request replaces the cached entry with a fresh LLM result.

Relationships

Table Relationships

  • users 1 → many workspaces A user owns many workspaces. workspaces.user_id → users.id (CASCADE DELETE). Soft-delete via deleted_at. At least one non-deleted workspace per user is enforced at the API layer.
  • workspaces 1 → many notes A workspace contains many notes. notes.workspace_id → workspaces.id (CASCADE DELETE). RAG retrieval is filtered by this FK.
  • workspaces 1 → many conversations A workspace contains many conversations. conversations.workspace_id → workspaces.id. Conversation list is filtered by active workspace.
  • workspaces 1 → many reminders A workspace contains many reminders. reminders.workspace_id → workspaces.id. Pending reminder count drives the workspace alert badge.
  • users 1 → many notes A user owns many notes (via user_id). notes.user_id → users.id (CASCADE DELETE). Soft-delete via deleted_at.
  • notes 1 → 0..1 embeddings Each note has at most one embedding. embeddings.note_id UNIQUE FK (CASCADE DELETE). Workspace scope is inherited through the note.
  • users 1 → many conversations A user owns many conversations (via user_id). conversations.user_id → users.id. Soft-delete via deleted_at.
  • conversations 1 → many messages A conversation contains many messages. messages.conversation_id → conversations.id (CASCADE DELETE)
  • users 1 → many reminders A user owns many reminders (AI-generated and manual). reminders.user_id → users.id (CASCADE DELETE)
  • users 1 → many feedback A user may submit many feedback entries. feedback.user_id → users.id (CASCADE DELETE). user_email is denormalised for admin display.
  • feature_flags standalone - feature_flags has no foreign keys. It is a global key-value configuration table not scoped to any user or workspace. All authenticated users can read; only admins can write.
  • system_logs standalone - system_logs has no foreign keys. user_id is stored as a plain integer so log entries survive user deletion. The table is global and admin-readable via GET /api/admin/logs. Admins can clear all logs via DELETE /api/admin/logs/clear.
  • translation_cache standalone - translation_cache has no foreign keys. It is a shared global cache keyed by SHA-256 hash of source_text::target_language. Not scoped to any user - all users benefit from and contribute to the same cache pool.
  • users 1 → 0..1 onedrive_accounts An admin connects at most one OneDrive account. onedrive_accounts.user_id UNIQUE FK → users.id (CASCADE DELETE). Tokens are Fernet-encrypted.
  • books 1 → many user_books A book can be borrowed by many users. user_books.book_id → books.id (CASCADE DELETE). UNIQUE on (user_id, book_id).
  • books 1 → many book_bookmarks, book_highlights A book can have many bookmarks and highlights across users. Both FKs CASCADE DELETE from books and from users.
  • notes 1 → many book_highlights Each highlight mirrors into one "{Book Title} - Highlights" note. book_highlights.note_id → notes.id (ON DELETE CASCADE) - deleting that note deletes all of its highlights.
  • books 1 → many book_note_sources A book can have one "Save as Note Source" job per user. book_note_sources.book_id → books.id (CASCADE DELETE). UNIQUE on (user_id, book_id).
  • book_note_sources 1 → many book_note_chunks A completed extraction job produces many chunked notes. book_note_chunks.book_note_source_id → book_note_sources.id (CASCADE DELETE); each chunk also links to notes.id and books.id (both CASCADE DELETE).

Migration

Migration Notes - Knowledge Workspace & Admin Features

The following changes are applied at application startup via ALTER TABLE ... IF NOT EXISTS or CREATE TABLE IF NOT EXISTS so existing deployments upgrade automatically without data loss.

📋 Startup migration steps Applied in order - idempotent
StepSQL OperationNotes
1 CREATE TABLE IF NOT EXISTS workspaces (...) Creates the workspaces table on first boot
2 CREATE UNIQUE INDEX IF NOT EXISTS idx_workspaces_user_name_uq ON workspaces(user_id, name) Enforces unique workspace names per user
3 ALTER TABLE notes ADD COLUMN IF NOT EXISTS workspace_id INTEGER REFERENCES workspaces(id) Existing notes get workspace_id = NULL initially; a backfill step assigns them to a default workspace
4 ALTER TABLE conversations ADD COLUMN IF NOT EXISTS workspace_id INTEGER REFERENCES workspaces(id) Same backfill pattern as notes
5 ALTER TABLE reminders ADD COLUMN IF NOT EXISTS workspace_id INTEGER REFERENCES workspaces(id) Same backfill pattern as notes
6 Backfill: for each existing user, create a default workspace named "My Workspace" (type: Other), then set workspace_id on all their orphaned rows Run once - safe to skip if workspace_id is already populated
7 CREATE INDEX IF NOT EXISTS for all new workspace-scoped indexes All indexes listed in the Indexes section above
8 ALTER TABLE users ADD COLUMN IF NOT EXISTS is_admin BOOLEAN DEFAULT false Adds the admin flag to the users table. Existing users get false; the first user is promoted via a startup check.
9 CREATE TABLE IF NOT EXISTS feature_flags (key VARCHAR(100) PRIMARY KEY, value TEXT NOT NULL, updated_at TIMESTAMPTZ DEFAULT now()) Creates the feature flags table. Default rows are inserted with INSERT ... ON CONFLICT DO NOTHING so existing customised values are preserved.
10 CREATE TABLE IF NOT EXISTS feedback (id SERIAL PRIMARY KEY, user_id INTEGER REFERENCES users(id), user_email VARCHAR NOT NULL, type VARCHAR(20) NOT NULL, message TEXT NOT NULL, status VARCHAR(20) DEFAULT 'open', created_at TIMESTAMPTZ DEFAULT now()) Creates the feedback table for bug reports and suggestions.
11 CREATE INDEX IF NOT EXISTS for feedback(user_id), feedback(status), feedback(type) Supports admin filtering queries on the Feedback tab.
12 First-user admin promotion: if no user has is_admin = true, set is_admin = true for the user with the lowest id Idempotent - safe to run on every startup; skipped if an admin already exists.
13 CREATE TABLE IF NOT EXISTS system_logs (id SERIAL PRIMARY KEY, created_at TIMESTAMPTZ NOT NULL DEFAULT now(), level VARCHAR(10) NOT NULL, source VARCHAR(100) NOT NULL, message TEXT NOT NULL, details JSONB, user_id INTEGER); CREATE INDEX IF NOT EXISTS ix_system_logs_created_at ON system_logs(created_at) Creates the system_logs table and its index. Standalone - no foreign keys. Applied on every startup; idempotent due to IF NOT EXISTS guards.
14 CREATE TABLE IF NOT EXISTS translation_cache (id SERIAL PRIMARY KEY, text_hash VARCHAR(64) UNIQUE NOT NULL, source_text TEXT NOT NULL, target_language VARCHAR(100) NOT NULL, translation TEXT NOT NULL, accuracy INTEGER, model VARCHAR(100) NOT NULL, hit_count INTEGER NOT NULL DEFAULT 0, created_at TIMESTAMPTZ NOT NULL DEFAULT now(), updated_at TIMESTAMPTZ NOT NULL DEFAULT now()); CREATE UNIQUE INDEX IF NOT EXISTS ix_translation_cache_text_hash ON translation_cache(text_hash) Creates the translation_cache table and its unique index. Standalone - no foreign keys. Applied on every startup; idempotent due to IF NOT EXISTS guards.

Risk (workspaces): Existing users with data but no workspace will have a synthetic "My Workspace" auto-created on first login. This is transparent to the user - they see their existing notes, conversations, and reminders already organised under one workspace and can rename or restructure from there.

Risk (admin): Existing deployments will automatically promote the user with the lowest ID to admin on the next startup. This is expected and intentional. Admins can immediately use the Admin Panel to promote or demote other users.

S3 pre-signed upload: The S3 upload feature (POST /api/upload/presign and POST /api/upload/process) requires no database changes. Uploaded files are processed transiently - Lambda downloads each S3 object, creates a note in the existing notes table, then deletes the S3 object. No new columns or tables are needed.

Evaluation & Survey

Evaluation & Survey Tables

These tables support the in-app mixed-method research evaluation framework. No raw prompts, AI answers, note content, uploaded files, or credentials are stored in any evaluation or survey table. Only metadata, IDs, counts, timings, ratings, and anonymised references are persisted.

The evaluation design also uses an explicit consent model. Participants are shown a short participant information statement before data collection starts, consent is versioned and timestamped, and evaluation participation can be withdrawn through an admin-reviewed withdrawal process that either deletes or anonymises the participant's evaluation records. This keeps the research workflow aligned with capstone ethics expectations while preserving pseudonymised analysis where permitted.
