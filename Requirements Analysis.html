Requirements Analysis

Functional and non-functional requirements, use cases, and user stories for MemoLink.

FR — Functional Requirements

Functional Requirements

ID Requirement Priority Category
FR-01 Users must be able to register with an email and password. High Auth
FR-02 Users must be able to log in with their credentials and receive a session token. High Auth
FR-03 Each user's notes and conversations must be isolated — users cannot access each other's data. High Auth
FR-04 Users must be able to create a new note with a title, body content, and optional source label. High Notes
FR-05 Users must be able to list all their notes. High Notes
FR-06 Users must be able to update the title and/or content of an existing note. Medium Notes
FR-07 Users must be able to delete a note. Deletion must cascade to the associated embedding. High Notes
FR-08 When a note is created or updated, the system must automatically generate and store a vector embedding for the note content. High AI/RAG
FR-09 The chat endpoint must embed the user's prompt and retrieve the top-K most semantically similar notes using cosine similarity. High AI/RAG
FR-10 The system must send the retrieved note context along with the user prompt to the LLM and return the generated answer. High AI/RAG
FR-11 Each chat response must include a list of source citations (note_id, title, snippet) for transparency. High AI/RAG
FR-12 Users must be able to upload files (PDF, DOCX, TXT) directly in the chat. The system must extract text, embed it, and include it in retrieval. High Files
FR-13 Users must be able to create a conversation session, and all messages in that session must be persisted. High Conversations
FR-14 Users must be able to list all their conversations. High Conversations
FR-15 Users must be able to rename or delete a conversation. Medium Conversations
FR-16 Conversation messages must be paginated — the client requests a limit and a cursor (before_id). Medium Conversations
FR-17 Users must be able to save a chat AI response directly as a new note. Medium Conversations
FR-18 Users must be able to delete individual messages from a conversation. Low Conversations
FR-19 The system must support streaming AI responses via Server-Sent Events (SSE). Tokens must reach the client incrementally with a blinking cursor in the UI. High AI/RAG
FR-20 The chat input must include a microphone button that captures voice input via the useRecording hook and transcribes it as the user's prompt. Medium Chat
FR-21 The system must provide a Reminders panel (right sidebar) supporting AI-generated and manual reminders. Each reminder has a title, description, due date, due time, type, and done flag. High Reminders
FR-22 AI reminder suggestions must be generated exclusively from the user's stored notes (up to 15, HTML-stripped), returning text, description, due_date, and due_time from GPT. High Reminders
FR-23 The note editor must support importing YouTube video captions via URL. The system fetches captions using youtube-transcript-api and inserts them as note content. A clear error is returned when no captions are available. High Video
FR-24 The note editor must support uploading audio/video files (MP4, M4A, WebM, MP3, WAV, MOV up to 25 MB) and transcribing them via OpenAI Whisper for use as note content. High Video
FR-25 Users must be able to change their password from the Settings panel by providing their current password and a new password (minimum 8 characters). The current password must be verified before the change is applied. Medium Auth
FR-26 The entire MemoLink stack must be deployable via docker compose up --build using only a populated .env file — no manual server configuration required. Medium Ops
FR-27 A new user who has no Knowledge Workspace must be redirected to a first-time workspace setup page immediately after login. The main dashboard must not be accessible until at least one workspace exists. High Workspaces
FR-28 The workspace creation form must collect: Workspace Name (required, unique per user), Workspace Type (Academic, Professional, Personal, Project, or Other), and an optional Description. High Workspaces
FR-29 After the first workspace is created, the system must mark it as the default workspace, set it as the active workspace in the user's session, and redirect the user to the main dashboard. High Workspaces
FR-30 Returning users must have their last selected workspace loaded automatically after login. The last_accessed_at timestamp on the workspace record is updated each time the user activates that workspace. High Workspaces
FR-31 A workspace selector dropdown must appear in the upper-left sidebar. It must display the active workspace name and allow the user to: switch to another workspace, create a new workspace, edit the active workspace's name/type/description, and view alert status. High Workspaces
FR-32 If a workspace has at least one pending (incomplete) reminder or unresolved alert, a visible badge or icon must appear beside its name in the workspace dropdown. Medium Workspaces
FR-33 When the active workspace changes, all visible content — notes list, conversation list, reminders panel, file uploads, and AI chat history — must update to show only data that belongs to the newly selected workspace. High Workspaces
FR-34 The RAG pipeline must restrict vector similarity search to notes and uploaded files that belong to the active workspace. Cross-workspace retrieval must not occur unless the user explicitly enables it. High Workspaces
FR-35 Each RAG source citation must include the workspace_id (or workspace name) of the note it came from, alongside the existing note_id, title, and snippet fields, to support traceability across workspaces when cross-workspace search is enabled. Medium Workspaces
FR-36 The system must not allow a user to delete their only remaining workspace. Soft-deletion is used — deleted workspaces are hidden from the dropdown but remain in the database. A workspace may only be soft-deleted when the user owns at least two active workspaces. High Workspaces
FR-37 Users must be able to request a password reset by submitting their email address. The system must send a signed JWT reset link (1-hour expiry, purpose:"reset" claim) to the user's registered email address via SMTP. When SMTP is not configured, the link must be logged to the console. The endpoint must always return HTTP 200 regardless of whether the email exists — to prevent email enumeration. High Auth
FR-38 Users must be able to reset their password by submitting a valid JWT reset token and a new password. The system must verify the token signature, expiry, and purpose:"reset" claim, then hash and store the new password using bcrypt. Returns 400 on expired token and 401 on invalid token. High Auth
FR-39 The chat interface must support AI image generation via DALL-E 3. Image generation is triggered by prompts starting with /image or natural-language phrases such as "generate an image of", "draw me a", "create a picture of", "paint", or "sketch". The system must call DALL-E 3 with size="1024x1024", quality="standard", and response_format="b64_json", and return the image as a persistent base64 data URL embedded in the chat message. High AI/Chat
FR-40 The Settings modal must allow users to select their preferred chat model from: GPT-4o Mini (default), GPT-4o, GPT-4 Turbo, GPT-3.5 Turbo, Gemini 2.0 Flash, Gemini 2.0 Flash Lite, Gemini 1.5 Flash 8B, and Gemini 1.5 Pro. Gemini models must be accessed via an OpenAI-compatible endpoint using GEMINI_API_KEY. When Gemini hits a RateLimitError during chat, the system must silently fall back to GPT-4o-mini and log a warning — no error must be shown to the user. High AI/Chat
FR-41 Every AI chat message must display a model attribution badge (e.g. "replied by GPT-4o Mini", "replied by DALL-E 3") below the action bar. The SSE done event must include a model field carrying the exact model identifier used for that response. Medium AI/Chat
FR-42 The translation endpoint must use a quality-improvement loop: (1) Gemini 2.0 Flash Lite performs the initial translation; (2) the result is back-translated to the source language; (3) GPT scores semantic similarity on a 0–100 scale; (4) if the score is below 85, the feedback is returned to Gemini for a refined translation. This loop repeats for up to 3 rounds. A 1.2-second sleep between API calls must be applied to avoid burst-rate detection. If Gemini is unavailable due to rate limits, the system must fall back to GPT-4o-mini for the full translation. Māori translation must inject a hint for macron usage (ā ē ī ō ū) and natural fluency. High Translation
FR-43 The translation bubble in the UI must display the model used and the final accuracy score (e.g. "by Gemini 2.0 Flash Lite · 87% accuracy"). The accuracy score must be colour-coded: green for ≥ 85, amber for ≥ 70, red for < 70. The translation response must include translation, accuracy (integer or null), and model fields. Medium Translation
FR-44 The bulk note upload endpoint (POST /api/notes/bulk) must preserve rich formatting from uploaded files. DOCX files must preserve headings (H1–H4 from paragraph style names), bold, italic, and underline inline runs, bullet and numbered lists, and tables (with <th> header rows), interleaved in document order. PDF files must infer headings by font-size ratio (≥ 1.5× body → H1; ≥ 1.3× → H2; ≥ 1.15× or bold+short → H3). PPTX files must map slide titles to <h2>, content items to <ul> with indent levels, and separate slides with <hr>. High Files
FR-45 When processing DOCX or PPTX files, the system must extract embedded images and include them inline as base64 <img> tags in the generated HTML. DOCX extraction must resolve XML blip relationship parts; PPTX extraction must detect picture shapes via shape.image. Medium Files
FR-46 The Tiptap note editor must support image insertion via three methods: (1) a toolbar button opening a local file picker; (2) drag-and-drop of image files onto the editor area; and (3) paste from clipboard (supporting screenshots and copied images). Images must render with rounded corners, a dark border, and a blue selection outline. The hasRichHtmlStructure() helper must recognise <img> tags as rich content. Medium Notes
FR-47 The chat input must include a web search toggle (globe icon). When enabled for a message, the system must query the Brave Search API using BRAVE_SEARCH_API_KEY and inject the top results as a system context block ([WEB SEARCH RESULTS for "..."]) before the LLM call. The web_search: bool field must be accepted by both POST /api/chat and POST /api/chat/stream. High AI/Chat
FR-48 The system must support an Agent mode accessible via an "Agent" toggle in ChatInput. When active, the frontend must call streamAgentChat() targeting POST /api/chat/agent/stream. The SSE stream must emit {tool_call, label} events as status chips during tool execution, {tool_result, ok} events upon tool completion, and then regular {t} token events for the final answer. Agent mode bypasses the standard RAG flow and uses tool-use capabilities. Medium AI/Chat
FR-49 The users table must include an is_admin BOOLEAN column. The first user to register must be automatically promoted to admin on application startup. JWT access tokens must include an is_admin claim that reflects the user's current admin status. A get_current_admin FastAPI dependency must verify the claim and return HTTP 403 if the authenticated user is not an admin. High Admin
FR-50 Admins must be able to promote or demote any other user's admin status via PATCH /api/admin/users/{id}/role. Admins must be able to list all registered users with their email, ID, and admin status via GET /api/admin/users. High Admin
FR-51 The system must maintain a feature_flags table with at least 8 entries: web_search_enabled, agent_mode_enabled, model_selection_enabled, image_generation_enabled, translation_enabled, file_upload_enabled, default_model (string), and default_language (string). These flags must be seeded with default values on startup. Admins must be able to read and update all flags via GET /api/admin/features and PUT /api/admin/features. Any authenticated user must be able to read the current flag values via GET /api/features. High Admin
FR-52 The frontend must fetch feature flags on login and apply them: disabled-feature buttons must be hidden or disabled; if model_selection_enabled is false the model selector must be hidden and the default_model flag value must be used for all requests; if default_language is set, the translation UI must default to that language. High Admin
FR-53 Any authenticated user must be able to submit a bug report or feature suggestion via POST /api/feedback. The request must include type (bug or suggestion) and a non-empty message. The submission must be stored in the feedback table with status="open", the authenticated user's ID and email, and a UTC created_at timestamp. An optional SMTP notification may be sent to the admin email on each new submission. Medium Feedback
FR-54 Admins must be able to list all feedback entries via GET /api/admin/feedback with optional query parameters type (bug, suggestion, or all) and status (open, read, resolved, or all). Admins must be able to update the status of any feedback entry via PATCH /api/admin/feedback/{id}. Non-admin users must receive HTTP 403 on both endpoints. Medium Feedback
FR-55 The Admin Panel must be accessible as a full-screen overlay from the user menu (avatar button) and must only be visible to admin users. It must contain three tabs: (1) Feedback — list and manage reports; (2) Feature Flags — toggle boolean flags and set string defaults; (3) Users — list all users and change admin status. Non-admin users must not see the Admin Panel option in the UI. Medium Admin
FR-56 The image generation pipeline must attempt models in the following fallback order: gpt-image-2 → gpt-image-1 → DALL-E 3 → DALL-E 2 → Pollinations.ai (free HTTP API, no API key). If all paid models fail, the Pollinations.ai URL-based endpoint must be called as a final fallback. Images must always be stored as base64 data URLs in messages.content — no file storage. Base64 image content must be stripped from the conversation history and RAG context before sending to OpenAI to prevent token overflow. An animated spinner must be shown in the UI while the image is generating. High AI/Chat
FR-57 The system must support a Research Mode accessible via a dedicated toggle in ChatInput. When active, the frontend must call streamResearch() targeting POST /api/research/stream. The backend must execute a four-step pipeline over SSE: (1) hybrid note retrieval — vector top-30 candidates re-ranked by keyword scoring; (2) optional live web search via Brave Search API; (3) academic paper search via OpenAlex (free, no key) or Semantic Scholar (with API key); (4) structured LLM synthesis. The SSE stream must emit {tool_call, label} and {tool_result, ok} events for each step so progress is shown in the UI. The final report must follow a fixed markdown structure: Research Summary, Key Findings (From Your Notes / From Web & Academic Sources), Knowledge Gaps, Contradictions & Tensions, Academic References, and Suggested Follow-up Research. High AI/Research
FR-58 The academic paper search component of Research Mode must use OpenAlex as the primary source (free, no API key, no rate limit). If a SEMANTIC_SCHOLAR_API_KEY is configured, Semantic Scholar must be used instead, with OpenAlex as a transparent fallback on failure. The Semantic Scholar integration must implement exponential backoff on HTTP 429 responses (1 s → 2 s → 4 s). When the user's prompt is a meta-question about their notes (containing words such as "gap", "notes", "review", "summarize"), the system must derive a search query from the first retrieved note title rather than sending the raw conversational prompt to the academic API. Medium AI/Research
FR-59 The Settings modal must allow users to select DeepSeek models in addition to OpenAI and Gemini: DeepSeek V3 (deepseek-chat) and DeepSeek R1 (deepseek-reasoner). DeepSeek models must be accessed via an OpenAI-compatible endpoint at https://api.deepseek.com/v1 using DEEPSEEK_API_KEY. The model selection UI must display DeepSeek models in a separate right-hand column to prevent the panel from overflowing vertically. Without DEEPSEEK_API_KEY configured, DeepSeek models must remain visible in the selector but will fail at the API layer with a clear error. Medium AI/Chat
FR-60 The system must be deployable to cloud platforms without code changes, using only environment variable configuration: the React/Vite frontend must deploy to Azure Static Web Apps via a GitHub Actions workflow (azure/static-web-apps-deploy@v1); the FastAPI backend must deploy as a Docker container to Railway (zero-config) or Azure Container Apps (via ACR build + az containerapp update); the database must remain on Supabase (no migration required). CI/CD must trigger on push to main with path filters so frontend and backend pipelines deploy independently. Medium Ops
FR-61 The system must support three user access levels — Regular, Plus, and Pro — stored as an access_level column on the users table (default: regular). The access level must be embedded in the JWT token at login/register and propagated in the TokenResponse. Admin users bypass all level checks regardless of their own access_level. High Auth
FR-62 The Admin Panel must allow administrators to: (1) view and change any user's access level (PATCH /api/admin/users/{id}/level) from the Users tab — each user row shows a colour-coded tier badge (Regular = grey, Plus = blue, Pro = purple) and a level dropdown; (2) configure a minimum access level per feature from the Feature Flags tab via a three-tab panel (Regular / Plus / Pro), where toggling a feature on/off for a given tab updates the feature's *_min_level flag. The Pro tab is read-only, always showing full access. Feature flags now include research_mode_enabled plus seven *_min_level flags (web_search_min_level, agent_mode_min_level, research_mode_min_level, image_generation_min_level, model_selection_min_level, translation_min_level, file_upload_min_level). High Admin
FR-63 The backend must enforce access level checks at the API layer — not just in the frontend UI. The GET /api/features endpoint must read the requesting user's access level from the JWT and return each feature as disabled if the user's level is below the configured *_min_level flag. The POST /api/research/stream endpoint must independently validate the user's level against research_mode_min_level and the research_mode_enabled global toggle, returning HTTP 403 if access is denied. Admin users always bypass level restrictions. High Auth
NFR — Non-Functional Requirements

Non-Functional Requirements

ID Requirement Priority Category
NFR-01 API response time for non-AI endpoints must be under 300 ms at p95 under normal load. High Performance
NFR-02 AI chat responses must complete within 10 seconds under normal conditions (bounded by LLM latency). Streaming must deliver the first token within 2 seconds. High Performance
NFR-03 No API keys, secrets, or passwords may be committed to version control. High Security
NFR-04 Passwords must be hashed before storage; plain-text credentials must never be persisted. High Security
NFR-05 The system must follow Clean Architecture — controllers must not contain business or persistence logic. High Maintainability
NFR-06 All services must implement interface protocols to support unit testing without a live database. Medium Testability
NFR-07 The System Usability Scale (SUS) score for the frontend must reach ≥ 68 (above-average) in evaluation. High Usability
NFR-08 The frontend must be accessible on desktop and mobile viewports without horizontal scroll. Medium Usability
NFR-09 RAG answers must cite at least one source note; uncited answers are considered a system failure. High Faithfulness
NFR-10 The system must be deployable with a single environment-variable file (.env) — no hard-coded configuration. Medium Portability
NFR-11 The system must be fully deployable via docker compose up with only .env configuration — no ffmpeg, yt-dlp, or other host-level tool installations required. Medium Portability
NFR-12 AI image generation via DALL-E 3 must complete and deliver the image to the user within 15 seconds under normal network conditions. The first streaming token ("🎨 Generating your image…") must appear within 1 second of the request. Medium Performance
NFR-13 Password reset tokens must expire after 1 hour. The system must reject expired or structurally invalid tokens with an appropriate HTTP error code. The reset endpoint must never reveal whether an email address is registered in the system. High Security
NFR-14 The translation quality loop must not block the request for longer than 30 seconds in total. The 1.2-second inter-call sleep and up to 3 refinement rounds must complete within this window; if Gemini fails due to rate limiting, GPT fallback must be invoked immediately. Medium Performance
NFR-15 All admin-only endpoints must return HTTP 403 for any request whose JWT does not carry is_admin: true. The is_admin claim must be verified server-side on every request — no client-side-only enforcement is acceptable. High Security
NFR-16 Base64-encoded image data embedded in message content must be stripped from all conversation history and RAG context arrays before they are sent to the OpenAI API, to prevent token overflow errors. The stripping operation must be transparent to the user — stored messages must retain the full base64 content for display. High Reliability
NFR-17 Feature flag reads must be served from a single feature_flags table row-set and must not require a full application restart when updated. Flag changes applied by an admin must be reflected to new user sessions without redeployment. Medium Maintainability
UC — Use Cases

Use Cases

Actor: New User
UC-01 · Register
User submits email + password. System validates uniqueness, hashes password, and creates account. Returns user record.
Actor: Registered User
UC-02 · Login
User submits credentials. System verifies hash. Returns user ID stored client-side for subsequent requests.
Actor: Logged-in User
UC-03 · Create Note
User writes a note in the Tiptap rich editor. System stores it and asynchronously generates + stores a vector embedding for retrieval.
Actor: Logged-in User
UC-04 · Upload File
User uploads a file in chat. System extracts text, embeds it as a temporary note, and includes it in retrieval context.
Actor: Logged-in User
UC-05 · Ask AI Question
User types a question. System embeds it, retrieves top-K notes, sends context to LLM, returns answer + citations. Tokens stream to the UI in real time.
Actor: Logged-in User
UC-06 · Browse Conversations
User views all past sessions, selects one, and paginates through its message history.
Actor: Logged-in User
UC-07 · Save AI Reply as Note
User selects an AI message and saves it as a new note with a custom title for future retrieval.
Actor: Logged-in User
UC-08 · Manage Notes
User views, edits title/content, or deletes notes from their library. Deletion cascades to the embedding.
Actor: Logged-in User
UC-09 · Rename / Delete Conversation
User renames a conversation for easier identification, or deletes it along with all its messages.
Actor: Logged-in User
UC-10 · Voice Input Chat
User taps the mic button in ChatInput. The useRecording hook captures audio, the system transcribes it, and inserts the text as the prompt — no typing required.
Actor: Logged-in User
UC-11 · Import YouTube Video
User pastes a YouTube URL in the Video Import modal. System fetches captions via youtube-transcript-api and inserts the transcript as note HTML content. Returns a 422 error if no captions are available.
Actor: Logged-in User
UC-12 · Upload Meeting Recording
User drags an MP4/M4A/WebM/MP3/WAV/MOV file (up to 25 MB) onto the Upload File tab. System sends it to OpenAI Whisper for transcription and inserts the result as note content.
Actor: Logged-in User
UC-13 · Generate Reminders from Notes
User clicks “Suggest” in the Reminders panel. System strips HTML, concatenates up to 15 notes, sends them to GPT, and returns AI-generated reminders with title, description, due date, and due time.
Actor: Logged-in User
UC-14 · Change Password
User opens Settings from the avatar dropdown menu, enters their current password and a new password (≥ 8 chars). System verifies the current password before applying the change.
Actor: New User
UC-15 · Create First Workspace
After registration or first login, user is redirected to the workspace setup page. User provides a name, selects a type (Academic, Professional, Personal, Project, or Other), and optionally adds a description. The system creates the workspace, marks it as default and active, then redirects to the dashboard.
Actor: Logged-in User
UC-16 · Switch Active Workspace
User clicks the workspace dropdown in the upper-left sidebar and selects a different workspace. All visible notes, conversations, reminders, and AI context immediately update to reflect the chosen workspace. The selection is persisted as the user's last accessed workspace.
Actor: Logged-in User
UC-17 · Create Additional Workspace
User selects "Create New Workspace" from the workspace dropdown. The creation form appears. On submission the new workspace is saved, set as the active workspace, and the dropdown updates to reflect the change.
Actor: Logged-in User
UC-18 · Edit Workspace
User selects "Manage Workspaces" from the dropdown and edits the name, type, or description of any of their workspaces. Changes are persisted and the dropdown label updates immediately if the active workspace is edited.
Actor: Logged-in User
UC-19 · Delete Workspace
User attempts to delete a workspace. The system rejects deletion if it is the user's only remaining workspace. On success, the workspace is soft-deleted, removed from the dropdown, and the user is switched to another active workspace.
Actor: Logged-in User
UC-20 · View Workspace Alerts
User opens the workspace dropdown. Workspaces with at least one pending reminder or unresolved alert display a badge. User can switch to the alerted workspace to review the outstanding items in the Reminders panel.
Actor: Unauthenticated User
UC-21 · Reset Forgotten Password
User clicks "Forgot password?" on the LoginPage, enters their email, and receives a signed JWT reset link via email. User opens the link, which loads the reset-password form pre-filled from the ?reset_token= URL param, enters a new password, and regains access. The token expires after 1 hour.
Actor: Logged-in User
UC-22 · Generate Image in Chat
User types "/image a mountain landscape at sunset". MemoLink detects the image intent, calls DALL-E 3, streams a generating-status token, then delivers the image as a base64 data URL embedded in the chat. Attribution shows "replied by DALL-E 3".
Actor: Logged-in User
UC-23 · Switch Chat Model
User opens Settings and selects "Gemini 2.0 Flash" from the model dropdown. All subsequent chat messages use that model. If Gemini hits a rate limit, MemoLink silently falls back to GPT-4o-mini and the attribution badge reflects the actual model used.
Actor: Logged-in User
UC-24 · Translate with Quality Score
User selects text in the chat and requests translation to Māori. MemoLink runs the quality loop (Gemini → back-translate → GPT score → refine up to 3×), then displays the translation with "by Gemini 2.0 Flash Lite · 91% accuracy" in a green badge.
Actor: Logged-in User
UC-25 · Search the Web in Chat
User toggles the globe icon in ChatInput and submits a question about a recent event. MemoLink queries Brave Search, injects the top results as system context, and the LLM generates an answer grounded in both notes and current web information.
Actor: Logged-in User
UC-26 · Bulk Upload Formatted Document
User uploads a DOCX report via the bulk-upload dialog. MemoLink extracts headings, bold text, lists, tables, and embedded images — preserving full formatting as HTML. The resulting note is stored with rich structure and embedded base64 images ready for retrieval.
Actor: Logged-in User
UC-27 · Insert Image into Note
User pastes a screenshot from the clipboard into the Tiptap editor. The image is inserted inline at the cursor position as a base64 data URL, rendered with rounded corners and a dark border. The note is saved with the image embedded in its HTML content.
Actor: Logged-in User
UC-28 · Use Agent Mode
User enables the "Agent" toggle in ChatInput and asks a complex multi-step question. The SSE stream shows tool-call status chips as the agent reasons through sub-tasks, then delivers the final answer. The user can see which tools were invoked via the tool_result events.
Actor: Admin User
UC-29 · Manage Feature Flags
Admin opens the Admin Panel from the user menu and navigates to the Feature Flags tab. Admin toggles "web_search_enabled" off. All users now see the web search globe icon hidden in ChatInput. Admin sets "default_model" to "gpt-4o" — users without model selection enabled will use that model.
Actor: Admin User
UC-30 · Triage Feedback
Admin opens the Feedback tab in the Admin Panel. Filters by status "open". Reads a bug report describing a UI issue. Marks it as "read", investigates, fixes it, then marks it "resolved". The full submission including user email and timestamp is shown.
Actor: Admin User
UC-31 · Manage User Roles
Admin opens the Users tab in the Admin Panel. Sees a list of all registered users with their emails and admin status. Promotes a trusted colleague to admin, granting them access to the Admin Panel and admin endpoints.
Actor: Logged-in User
UC-32 · Submit Bug Report
User encounters a UI issue. Opens the Help modal and selects "Report a Bug". Fills in a description and submits. The report is stored in the database with status "open". An SMTP notification is sent to the admin (if configured). The user receives a confirmation that the report was submitted.
Actor: Logged-in User (Plus or Pro)
UC-33 · Use Research Mode
A Plus or Pro user enables the Research Mode toggle in ChatInput and submits a query. The frontend calls POST /research/stream. The backend checks the user's JWT access_level against the research_mode_min_level feature flag. If the check passes the streaming research pipeline runs and returns a cited, multi-source answer. If the user's level is below the minimum threshold the API returns HTTP 403 and the UI displays an access-denied message.
Actor: Admin User
UC-34 · Manage User Access Levels
Admin opens the Admin Panel and navigates to the Users tab. Each user row displays a colour-coded access level badge (Regular / Plus / Pro) and a dropdown to change it. Admin upgrades a user to Pro; the backend calls PATCH /admin/users/{id}/level and persists the new value. The user must re-login so their JWT reflects the updated level. In the Feature Flags tab the admin uses the tabbed Access Level Requirements UI to set research_mode_min_level to "plus", ensuring Regular users cannot access Research Mode until upgraded.
US — User Stories

User Stories

US-01   As a student, I want to upload my lecture notes so that I can ask AI questions about them during revision.
US-02   As a knowledge worker, I want AI answers to cite the exact note they came from so that I can verify the source before acting on the information.
US-03   As a user, I want to create an account so that my notes are private and only accessible to me.
US-04   As a researcher, I want to save a useful AI response back as a note so that it becomes part of my searchable knowledge base.
US-05   As a student, I want to attach a PDF directly in the chat so that I can immediately ask questions about it without creating a note first.
US-06   As a user, I want to browse my conversation history and scroll back through old messages so that I can reference what the AI previously told me.
US-07   As a power user, I want to delete notes I no longer need so that irrelevant content stops appearing in my AI answers.
US-08   As a user, I want to rename my conversations so that I can identify them at a glance in the sidebar.
US-09   As a user on a slow connection, I want AI responses to stream token-by-token so that I can start reading the answer before it has fully arrived.
US-10   As a student in a lecture, I want to speak my question aloud so that I can interact with the AI without interrupting my note-taking.
US-11   As a professional, I want to import a YouTube tutorial URL so that its transcript is instantly searchable in my knowledge base.
US-12   As a remote worker, I want to upload a Zoom recording so that the meeting content is transcribed and available for AI retrieval.
US-13   As a busy student, I want the system to suggest reminders from my notes automatically so that I do not miss deadlines embedded in my study material.
US-14   As a student, I want to keep my Capstone Research notes in a separate workspace from my Math notes so that AI answers never mix context from different subjects.

Constraints & Assumptions

Constraints & Assumptions

🔑

OpenAI API Required

Both embedding generation and answer generation depend on external OpenAI API access. Offline use is not supported.

🗄️

pgvector Extension

The database must have the pgvector extension enabled. Supabase provides this out of the box.

👤

Single-Tenant Model

MemoLink is designed for individual users. Team or multi-tenant sharing is out of scope for this capstone version.

📄

File Type Scope

Chat uploads: plain text (.txt), PDF (.pdf), Word (.docx). Video uploads: MP4, M4A, WebM, MP3, WAV, MOV (max 25 MB).

🧪

Evaluation Data

Evaluation uses only student-owned, publicly available, or participant-approved non-sensitive data. No private data.

🌐

Browser-Based Only

The frontend is a web SPA. Native mobile apps are out of scope for this version.

🎬

YouTube Caption Dependency

Video import via YouTube URL requires the video to have captions (auto-generated or manual). Videos without captions return a 422 error. No ffmpeg or yt-dlp is used.

📤

25 MB Upload Limit

Audio and video file uploads are capped at 25 MB — the OpenAI Whisper API limit. Files exceeding this are rejected before transcription.

🗂️

Workspace Minimum

Every user must have at least one workspace at all times. The system blocks deletion of the last remaining workspace and always enforces workspace-scoped data access at the API layer.

🔒

Workspace Ownership

Workspaces are strictly user-owned. Cross-user workspace access is not permitted. Team or workspace-sharing is out of scope for this capstone version.

🛡️

Single Admin Bootstrapping

Admin status is bootstrapped by auto-promoting the first registered user. There is no separate admin registration flow. Admins can promote additional users via the Admin Panel.

💬

Feedback Storage Only

Bug reports and suggestions are stored in the database and optionally emailed to the admin via SMTP. No external ticketing system integration (e.g. GitHub Issues, Jira) is provided in this version.
