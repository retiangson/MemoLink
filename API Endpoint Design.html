Auth

Authentication Endpoints

POST /api/v1/auth/register Register a new user account ▼

Creates a new user. Returns 400 if the email is already registered. Password is hashed with bcrypt (passlib) before storage.

Request Body - RegisterDTO
{
  "email":    "user@example.com",
  "password": "securepassword"
}
Response 200 - UserResponse
{
  "id":    1,
  "email": "user@example.com"
}
Error Responses
400 Email already registered
422 Validation error (missing fields)
POST /api/v1/auth/login Log in and return user record ▼

Verifies email + hashed password. Returns the user record whose id the client stores in localStorage key memolink_user for scoped requests. Returns 401 on invalid credentials.

Request Body - LoginDTO
{
  "email":    "user@example.com",
  "password": "securepassword"
}
Response 200 - UserResponse
{
  "id":    1,
  "email": "user@example.com"
}
POST /api/v1/auth/change-password Change password with current-password verification ▼

Verifies the supplied current_password against the stored bcrypt hash before updating. The new password must be at least 8 characters. Returns 401 if current_password is wrong, 400 if the new password fails length validation.

Request Body - ChangePasswordDTO
{
  "user_id":         1,
  "current_password": "oldpass123",
  "new_password":     "newpass456"
}
Response 200
{ "ok": true }
Error Responses
401 Current password incorrect
400 New password too short (<8 chars)
POST /api/v1/auth/forgot-password Request a password-reset email link ▼

Accepts an email address and, if a matching account exists, generates a signed JWT reset token with a 1-hour expiry and a purpose:"reset" claim. The reset link (FRONTEND_URL/login?reset_token=<token>) is sent to the user's address via SMTP. If SMTP is not configured the link is printed to the server console. The endpoint always returns HTTP 200 regardless of whether the email is registered - this prevents email enumeration attacks.

Request Body - ForgotPasswordDTO
{
  "email": "user@example.com"
}
Response 200 (always)
{
  "message": "If that email is registered, a reset link has been sent."
}
POST /api/v1/auth/reset-password Reset password using a JWT reset token ▼

Verifies the supplied JWT token (signature, expiry, and purpose:"reset" claim). On success, hashes the new password with bcrypt and persists it. Returns 400 if the token has expired, or 401 if the token is structurally invalid or the purpose claim does not match.

Request Body - ResetPasswordDTO
{
  "token":        "eyJhbGciOiJIUzI1NiIsInR5...",
  "new_password": "newSecurePass123"
}
Response 200
{
  "message": "Password reset successful."
}
Error Responses
400 Token has expired (past 1-hour window)
401 Invalid token signature or wrong purpose claim

POST /api/v1/notes/bulk Bulk upload files - rich formatting preserved ▼

Accepts one or more file uploads (DOCX, PDF, PPTX, TXT) as multipart/form-data and creates one note per file. Uses extract_formatted_html() instead of plain text extraction to preserve rich document structure:

DOCX - headings (H1–H4 from paragraph style names), bold/italic/underline inline runs, bullet and numbered lists, tables (with <th> header rows), embedded images as base64 <img> tags; paragraphs and tables interleaved in document order.
PDF - word-level extraction via pdfplumber; headings inferred by font-size ratio (≥ 1.5× body → H1; ≥ 1.3× → H2; ≥ 1.15× or bold+short → H3); bold text detected by fontname.
PPTX - slide title → <h2>; content items → <ul> with indent levels; slide separator → <hr>; picture shapes extracted as base64 <img> tags.

Each created note is automatically embedded for RAG retrieval.

Form Fields
user_id      integer  (required)
workspace_id integer  (optional)
files        file[]   (required, multipart)
Accepted: .docx .pdf .pptx .txt
Response 200 - NoteResponseDTO[]
[
  {
    "id":      60,
    "title":   "report.docx",
    "content": "<h1>Executive Summary</h1><p>...</p>",
    "source":  "report.docx"
  }
]
Chat / RAG

Chat Endpoints

POST /api/v1/chat Submit prompt - RAG answer + citations (JSON) ▼

Core RAG endpoint. Flow: 1 - Embed the prompt via OpenAI text-embedding-3-small. 2 - Retrieve top-K notes by cosine similarity (workspace-scoped). 3 - Optionally query Brave Search API and inject results as system context (when web_search: true). 4 - Detect image-generation intent and call DALL-E 3 if triggered. 5 - Call the selected model (GPT or Gemini) for the answer. 6 - Return answer + source citations + model used. Persists both the user message and the AI reply to the conversation. For real-time streaming use /chat/stream instead.

Request Body - ChatRequestDTO
{
  "user_id":         1,
  "conversation_id": 7,
  "prompt":          "What is retrieval-augmented generation?",
  "top_k":           5,
  "model":           "gpt-4o-mini",
  "web_search":      false
}
Response 200 - ChatResponseDTO
{
  "answer": "RAG combines retrieval...",
  "sources": [
    {
      "note_id": 42,
      "title":   "Lecture 5",
      "snippet": "RAG stands for..."
    }
  ],
  "attachments": [],
  "message_id": 103,
  "model": "gpt-4o-mini"
}
POST /api/v1/chat/stream Streaming AI response via SSE (text/event-stream) ▼

Same RAG pipeline as /chat but returns Content-Type: text/event-stream. The selected model is called with stream=True. Each token is forwarded immediately. Supports optional web search context injection (web_search: true) and multi-model selection via the model field. When an image-generation intent is detected, yields "🎨 Generating your image…" first, then the base64 image tag. nginx must have proxy_buffering off and X-Accel-Buffering: no - both are set in nginx.conf. The frontend displays a blinking cursor while streaming; the cursor disappears when done is received.

Request Body - ChatRequestDTO
{
  "user_id":         1,
  "conversation_id": 7,
  "prompt":          "Explain embeddings.",
  "top_k":           5,
  "model":           "gemini-2.0-flash",
  "web_search":      true
}
SSE Event Stream
Content-Type: text/event-stream

data: {"t":"Embed"}
data: {"t":"dings"}
data: {"t":" are"}
... one event per token ...
data: {"done":true,"id":104,"model":"gemini-2.0-flash"}
POST /api/v1/chat/upload Upload files + prompt in one multipart request ▼

Accepts multipart/form-data. Extracts text from uploaded files (PDF, DOCX, TXT), embeds them as temporary notes, then runs the full RAG pipeline with the combined context. Attachments are returned in the response.

Form Fields
conversation_id integer  (required)

files            file[]   (required, multipart)
Accepted: .txt, .pdf, .docx
Response 200 - ChatResponseDTO
{
  "answer": "Based on your document...",
  "sources": [...],
  "attachments": [
    {
      "filename":     "report.pdf",
      "content_type": "application/pdf",
      "size":         204800
    }
  ],
  "message_id": 104
}
POST /api/v1/chat/agent/stream Agentic chat with tool-call SSE events ▼

Agentic variant of the streaming chat endpoint. Bypasses the standard RAG pipeline and uses tool-use capabilities. The SSE stream emits three event types in sequence: (1) {"tool_call": "<name>", "label": "<display text>"} - rendered as status chips while a tool is executing; (2) {"tool_result": "<result>", "ok": true} - signals tool completion; (3) regular {"t": "<token>"} events for the final answer. The frontend "Agent" toggle switches streamChat() to streamAgentChat() which targets this endpoint.

Request Body - AgentChatDTO
{
  "conversation_id": 7,
  "prompt":          "Search and summarise the latest on RAG.",
  "workspace_id":   1,
  "model":           "gpt-4o-mini"
}
SSE Event Stream
Content-Type: text/event-stream

data: {"tool_call":"web_search","label":"Searching the web..."}
data: {"tool_result":"...","ok":true}
data: {"t":"Here"}
data: {"t":" is"}
... token events ...
data: {"done":true,"id":110,"model":"gpt-4o-mini"}

Translate

Translation Endpoint

POST /api/v1/translate Translate text to a target language ▼

Translates text using a multi-round quality loop: (1) Gemini 2.0 Flash Lite performs the initial translation; (2) the result is back-translated to the source language; (3) GPT scores semantic similarity (0–100); (4) if the score is below 85, the feedback is fed back to Gemini for a refined translation - this repeats for up to 3 rounds. A 1.2-second sleep is applied between API calls to avoid burst-rate detection. If Gemini hits a rate limit the entire pipeline falls back to GPT-4o-mini. For Māori translations a hint is injected specifying correct macron usage (ā ē ī ō ū) and natural fluency. The accuracy field in the response is the final similarity score; it is null when the fallback model is used without a scoring round. Results are cached in the translation_cache table - repeated requests for the same text and language are served instantly with "cached": true. The optional force boolean field (default false) bypasses the cache: when true, the full LLM pipeline runs and the cache entry is updated with the fresh result.

Request Body - TranslateDTO
{
  "text":            "Hola mundo",
  "target_language": "English",
  "force":           false
}
Response 200
{
  "translation": "Hello world",
  "accuracy":    92,
  "model":       "gemini-2.0-flash-lite",
  "cached":      false
}
Accuracy colour-coding in UI
≥ 85  Green  - high confidence
≥ 70  Amber  - acceptable
< 70   Red    - low confidence
null   No score (GPT fallback path)

Upload

S3 Pre-signed Upload Endpoints

Two-step flow that bypasses AWS Lambda's 6 MB payload limit. The client requests a pre-signed URL, uploads directly to S3, then submits the S3 key for server-side processing.

POST /api/v1/upload/presign Generate S3 pre-signed upload URL ▼

Validates that the file size is ≤200 MB and that S3_UPLOAD_BUCKET is configured. Generates an S3 pre-signed PUT URL with a 1-hour TTL (3600 seconds) scoped to the authenticated user's folder. The client uploads file bytes directly to S3 using an HTTP PUT request to the returned URL - Lambda is never in the upload path so the 6 MB Lambda payload limit does not apply. Auth: Bearer token required.

Request Body - PresignDTO

  "filename":     "recording.mp4",
  "content_type": "video/mp4",
  "size_bytes":   104857600
}
Response 200 - PresignResponse
{
  "url":        "https://s3.amazonaws.com/...",
  "key":        "uploads/1/abc/recording.mp4",
  "expires_in": 3600

Error Responses

503 S3_UPLOAD_BUCKET not configured

Commands

Slash Command Endpoint

Executes a slash command typed in the chat input. All 14 commands stream responses via SSE in the same format as /chat/stream. Commands are scoped to the authenticated user and active workspace.

POST /api/v1/commands/execute Execute a slash command - SSE stream ▼

Parses and executes a slash command. The command field contains the full typed text (e.g. /Improve "Assessment 2 Notes"). The backend saves the command as a user message, executes the operation (AI call, DB update, reminder creation, etc.) and streams the result.

Special SSE events beyond standard {"t":"token"}:
{"cmd_running":"message"} - shown while the AI is processing (dual-ring spinner in UI)
{"note_updated":id} - triggers frontend to refresh the note in sidebar and editor
{"quiz":{...}} - renders an interactive quiz component in the chat bubble
{"open_note":id} - opens the note in the editor (used by /Read)
{"speak":"plain text"} - triggers browser text-to-speech via Web Speech API (used by /Read; Markdown symbols stripped server-side before sending)

Request Body - SlashCommandRequestDTO
{
  "command":         "/Improve \"Assessment 2 Notes\"",
  "conversation_id": 7,
  "workspace_id":   1,
  "model":           "gpt-4o-mini"
}
SSE Event Stream
data: {"cmd_running":"Improving note..."}
data: {"note_updated":42}
data: {"t":"**Note** has been improved."}
data: {"done":true,"id":110,"conversation_id":7}

// /Read command example:
data: {"open_note":42}
data: {"t":"## My Note\n\nContent..."}
data: {"speak":"My Note. Content without markdown symbols."}
data: {"done":true,"id":111,"conversation_id":7}
