Deployment Guide

From local Docker Compose to production cloud. This guide covers every path to get MemoLink running — on your laptop, on a single VPS, or fully distributed across Azure Static Web Apps, Railway, and Supabase.

Overview

Deployment Options at a Glance

MemoLink's three-tier architecture maps cleanly to three hosting targets. Mix and match any combination — they communicate only over HTTP and environment variables.

🗄️ Database

Supabase — hosted PostgreSQL with pgvector. Already provisioned for this project. No additional setup required for any deployment path.

⚡ Backend

Railway (recommended) or Azure Container Apps. Both run the FastAPI container from the root Dockerfile with environment variables injected at runtime.

🌐 Frontend

Azure Static Web Apps (recommended) or any static CDN. The React/Vite build outputs a plain dist/ folder — no Node runtime required in production.

Option 1 — Local / Self-Hosted

Docker Compose (Single Machine)

The fastest path. One command starts the FastAPI backend and the React/nginx frontend together. Best for local development, demos, or a personal VPS.

1

Install Docker Desktop

Download from docker.com/products/docker-desktop (macOS / Windows / Linux). Make sure docker compose version returns v2.x or newer.

2

Create your .env file

Copy the template and fill in your secrets. The backend container reads this file at startup.

cp .env.example .env
# open .env and set at minimum:
# DATABASE_URL, OPENAI_API_KEY, SECRET_KEY
3

Build and start

docker compose up --build

The frontend container is served at http://localhost:80. nginx proxies all /api/* requests to the backend on port 8000. SSE streaming works because nginx is configured with proxy_buffering off.

4

Run in the background

docker compose up -d --build   # detached
docker compose logs -f         # follow logs
docker compose down            # stop
nginx configuration — memolink_web/nginx.conf sets proxy_buffering off, proxy_cache off, and proxy_read_timeout 300s so that AI streaming responses (SSE) pass through without buffering delays. Do not remove these directives.
Option 2 — Cloud (Recommended)

Cloud Deployment

Deploy the frontend and backend independently to managed platforms. Each service scales on its own, and the only coupling is the VITE_API_BASE_URL environment variable the frontend uses to reach the backend.

Recommended

Frontend → Azure Static Web Apps

Free tier, global CDN, GitHub Actions CI/CD built in. Supports custom domains and automatic HTTPS. The React/Vite build (dist/) is uploaded directly — no server runtime needed.

Recommended

Backend → Railway

Zero-config container deployments. Push to GitHub → Railway builds from the root Dockerfile automatically. Free trial included; production plans are usage-based (scales to zero when idle).

Azure-Native Alternative

Backend → Azure Container Apps

Serverless containers on Azure. Integrates with Azure Container Registry. Scales to zero. Better choice if you want everything in one Azure subscription (RBAC, billing, monitoring).

Frontend Deployment

Azure Static Web Apps

Azure Static Web Apps provides a free global CDN for static files with automatic CI/CD from GitHub. The React/Vite project (memolink_web/) deploys with a single GitHub Actions workflow file.

1

Create the Static Web App resource

In the Azure Portal, search for Static Web Apps → Create. Select your GitHub repo and set the build configuration: 

App location:    /memolink_web
Output location: dist
Build command:   npm run build
2

Set the VITE_API_BASE_URL environment variable

In the Static Web App resource → Configuration → Application settings, add one entry: 

VITE_API_BASE_URL = https://<your-backend-url>/api

This variable is injected at build time. It tells the React app where to send API requests. Set it to your Railway or Azure Container Apps backend URL.

3

Add GitHub Actions workflow

Azure creates this file automatically when you link your repo. If you need to add it manually, create .github/workflows/azure-static-web-apps.yml: 

name: Azure Static Web Apps CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Deploy to Azure Static Web Apps
        uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_API_TOKEN }}
          repo_token: ${{ secrets.GITHUB_TOKEN }}
          action: upload
          app_location: /memolink_web
          output_location: dist
        env:
          VITE_API_BASE_URL: ${{ secrets.VITE_API_BASE_URL }}

Add AZURE_STATIC_WEB_APPS_API_TOKEN and VITE_API_BASE_URL as GitHub repository secrets (Settings → Secrets and variables → Actions).

4

Push to main to deploy

Every push to main triggers a build and deploy. The live URL is shown in the Azure Static Web Apps resource overview (e.g. https://<random>.azurestaticapps.net). Custom domains are configured under Custom domains in the same resource.

Backend Deployment — Recommended

Railway

Railway detects the root Dockerfile and builds it automatically. No additional configuration files are required.

1

Create a new project

Go to railway.app → New Project → Deploy from GitHub repo. Select your MemoLink repository.

2

Set environment variables

In the service settings → Variables, add all keys from .env.example. At minimum:

DATABASE_URL       = postgresql+psycopg2://...
OPENAI_API_KEY     = sk-...
SECRET_KEY         = <random 32+ char string>
FRONTEND_ORIGINS   = https://<your-azure-static-url>
3

Configure the start command

Railway reads CMD from the Dockerfile by default:

uvicorn memolink_backend.main:app --host 0.0.0.0 --port 8000

No changes needed — the Dockerfile already sets this command.

4

Generate a public domain

In the service → Settings → Networking → Generate Domain. Copy the URL (e.g. https://memolink-production.up.railway.app) and use it as the VITE_API_BASE_URL in the frontend (append /api).

SSE streaming note — Railway does not buffer responses by default, so Server-Sent Events (AI token streaming) work without any extra configuration.
Backend Deployment — Azure-Native

Azure Container Apps

Use Azure Container Apps if you want everything in one Azure subscription. The workflow pushes the Docker image to Azure Container Registry, then deploys it to a Container App that scales to zero when idle.

1

Create Azure resources

# Resource group
az group create --name memolink-rg --location eastus

# Container Registry (to store Docker images)
az acr create --resource-group memolink-rg \
              --name memlinkacr --sku Basic

# Container Apps environment
az containerapp env create \
  --name memolink-env \
  --resource-group memolink-rg \
  --location eastus
2

Build and push the Docker image

# Build locally and push to ACR
az acr build --registry memlinkacr \
             --image memolink-backend:latest .
3

Deploy the Container App

az containerapp create \
  --name memolink-backend \
  --resource-group memolink-rg \
  --environment memolink-env \
  --image memlinkacr.azurecr.io/memolink-backend:latest \
  --registry-server memlinkacr.azurecr.io \
  --target-port 8000 \
  --ingress external \
  --min-replicas 0 \
  --max-replicas 3 \
  --env-vars \
    DATABASE_URL=secretref:database-url \
    OPENAI_API_KEY=secretref:openai-api-key \
    SECRET_KEY=secretref:secret-key
4

Automate with GitHub Actions

Create .github/workflows/deploy-backend.yml to rebuild and redeploy the Container App on every push to main:

name: Deploy Backend to Azure Container Apps

on:
  push:
    branches: [main]
    paths:
      - memolink_backend/**
      - requirements.txt
      - Dockerfile

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Azure Login
        uses: azure/login@v2
        with:
          creds: ${{ secrets.AZURE_CREDENTIALS }}

      - name: Build and push image to ACR
        run: |
          az acr build \
            --registry memlinkacr \
            --image memolink-backend:${{ github.sha }} \
            --image memolink-backend:latest .

      - name: Update Container App
        run: |
          az containerapp update \
            --name memolink-backend \
            --resource-group memolink-rg \
            --image memlinkacr.azurecr.io/memolink-backend:${{ github.sha }}

Store AZURE_CREDENTIALS (output of az ad sp create-for-rbac) as a GitHub secret.

Reference

Dockerfiles

Both Dockerfiles are already included in the repository.

Backend — Dockerfile (root)

FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["uvicorn", "memolink_backend.main:app", "--host", "0.0.0.0", "--port", "8000"]

Frontend — memolink_web/Dockerfile

Multi-stage build: Node compiles the Vite bundle, then nginx serves it.

# Stage 1: build
FROM node:20-slim AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
ARG VITE_API_BASE_URL=/api
ENV VITE_API_BASE_URL=$VITE_API_BASE_URL
RUN npm run build

# Stage 2: serve
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Compose — docker-compose.yml

services:

  backend:
    build:
      context: .
      dockerfile: Dockerfile
    env_file: .env
    restart: unless-stopped
    expose:
      - "8000"

  frontend:
    build:
      context: ./memolink_web
      dockerfile: Dockerfile
    ports:
      - "80:80"
    depends_on:
      - backend
    restart: unless-stopped
Configuration

Environment Variables Reference

All variables are read from .env (Docker Compose) or from the platform's secret/environment configuration (Railway / Azure Container Apps). Never commit the .env file to version control.

Variable Description Required?
DATABASE_URL PostgreSQL connection string, e.g. postgresql+psycopg2://user:pass@host/db. Supabase connection pooler URL. Required
OPENAI_API_KEY OpenAI API key. Used for GPT chat, embeddings (text-embedding-3-small), Whisper transcription, and DALL-E image generation. Required
SECRET_KEY Random string (32+ characters) used to sign JWT access tokens. Generate with openssl rand -hex 32. Required
FRONTEND_ORIGINS Comma-separated CORS origins allowed to call the API, e.g. https://your-app.azurestaticapps.net. Defaults to http://localhost:5173 if unset. Required
GOOGLE_API_KEY Google AI Studio key for Gemini models (translation quality loop, alternative chat models). Optional
BRAVE_SEARCH_API_KEY Brave Search API key. Required for the Web Search toggle in ChatInput. Obtain from api.search.brave.com. Optional
SMTP_HOST SMTP server hostname for outgoing email (password reset links, bug report notifications). Optional
SMTP_PORT SMTP port. Typically 587 (STARTTLS) or 465 (SSL). Optional
SMTP_USER SMTP login username (usually the sender email address). Optional
SMTP_PASSWORD SMTP login password or app-specific password. Optional
SMTP_FROM Sender email address shown in outgoing MemoLink emails. Optional
VITE_API_BASE_URL Frontend build-time variable. Full URL of the backend API including /api suffix, e.g. https://memolink.up.railway.app/api. Set in platform settings or memolink_web/.env for local dev. Required (frontend)
Cross-Origin

CORS Configuration

The FastAPI backend reads FRONTEND_ORIGINS to build the allowed-origins list for the CORS middleware. When the frontend and backend are on different domains (the typical cloud setup), this variable must be set correctly or all browser requests will be blocked. 

# .env — cloud deployment example
FRONTEND_ORIGINS=https://<random>.azurestaticapps.net,https://memolink.yourdomain.com
After adding a custom domain to Azure Static Web Apps, remember to also add it to FRONTEND_ORIGINS in the backend's environment variables and redeploy the backend service. Both the azurestaticapps.net subdomain and any custom domain need to be listed if both are in use.
Database

Supabase (No Additional Setup Required)

MemoLink's database is already hosted on Supabase with the pgvector extension enabled. The backend runs schema migrations automatically at startup using ALTER TABLE … IF NOT EXISTS and CREATE TABLE … IF NOT EXISTS statements — no manual migration tool is needed.

Tables auto-created on startup

users, conversations, messages, notes, embeddings, workspaces, reminders, feedback, feature_flags

Seeded defaults

Eight feature flags are inserted with ON CONFLICT DO NOTHING on every startup. The first registered user is automatically promoted to admin.

Connection string format

Use the Supabase connection pooler (Transaction mode) URL to avoid hitting the direct connection limit:
postgresql+psycopg2://postgres.<ref>:<pw>@aws-0-region.pooler.supabase.com:6543/postgres

pgvector extension

Already enabled on the Supabase project. The embeddings table stores 1536-dimensional vectors (text-embedding-3-small output) for cosine similarity search.

Go-Live

Post-Deployment Checklist

  • Backend health check passes: GET /api/health returns {"status":"ok"}
  • FRONTEND_ORIGINS includes the exact frontend URL (no trailing slash)
  • VITE_API_BASE_URL is set to the backend URL + /api in the frontend build
  • Register an account — confirm the first user is auto-promoted to admin (check via user menu)
  • Send a test chat message and verify AI streaming (SSE) works end-to-end
  • Upload a test note file and confirm it appears in the notes list with embeddings
  • Test the Admin Panel: toggle a feature flag and confirm it takes effect on the next page load
  • If SMTP is configured: trigger a password reset and confirm the email arrives
  • Verify Web Search toggle works (requires BRAVE_SEARCH_API_KEY)
  • Verify Gemini models are selectable (requires GOOGLE_API_KEY)
  • Confirm no .env file or secrets appear in the GitHub repository
