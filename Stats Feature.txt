You are working on the MemoLink project.

Task:
Implement a quantitative evaluation analytics module inside MemoLink.

Goal:
The app must automatically collect measurable research/evaluation data during prototype testing, then generate reports, charts, summaries, and CSV exports for Assessment 2 and later Assessment 3.

Important:
This is not the qualitative survey. The qualitative survey is separate.
This task is for quantitative data collection, system telemetry, measurable metrics, report dashboard, graphs, and exportable evaluation evidence.

Do not break existing features:
- normal chat
- RAG
- notes
- workspaces
- reminders
- transcription
- translation
- slash commands
- quiz/study mode
- timeline
- confidence level
- feature flags
- admin panel
- system logs
- authentication
- conversations
- user API keys
- Gmail integration
- existing feedback/bug report system

Main concept:
Create a dedicated “Evaluation Analytics” system separate from normal system_logs and separate from feedback/bug reports.

system_logs = debugging and operational logs.
feedback = bug reports and suggestions.
evaluation analytics = research data for measurable capstone evaluation.

Do not rely only on system_logs because they are not structured enough for research reporting.

Add a feature flag:
evaluation_analytics_enabled = true/false
evaluation_survey_enabled = true/false
evaluation_admin_export_enabled = true/false

All analytics collection must be disabled when evaluation_analytics_enabled = false.

Privacy and ethics requirements:
- Add an evaluation consent screen before starting an evaluation session.
- Do not collect real names.
- Use participant_code such as P001, P002, P003.
- Do not store full private note content, full prompts, full AI answers, or uploaded files in analytics tables.
- Store IDs, counts, timings, ratings, hashes, and short non-sensitive metadata only.
- If prompt/content reference is needed, store message_id or note_id, not the full content.
- Optional: store prompt_length_chars and answer_length_words instead of raw text.
- Do not expose analytics data to normal users.
- Admin only can view/export reports.
- Ensure all analytics records are scoped to user_id and workspace_id where applicable.
- Do not include API keys, secrets, stack traces, raw Gmail content, or private file contents in analytics.
- Add clear wording: “This evaluation records performance metrics and user ratings for academic assessment. It does not store your private note content inside the analytics dataset.”
